# Part 7 / Clip 5: 위키 개선 — 하위 인덱스에서 하이브리드 RAG로

## 이 클립에서 만드는 것

지난 Clip 4에서 본인 위키에 1,000편 규모의 원본 기사를 채우고 질문을 던져봤어요. 그 과정에서 평면 `index.md` 한 장의 한계가 드러났을 겁니다. 무슨 질문이든 전체 목록부터 읽어야 했고, `Dune`이 영화인지 시리즈인지 매번 헷갈렸죠. 한계는 지난 클립에서 이미 봤으니, 오늘은 그걸 **직접 고쳐서 작동시키는** 클립입니다.

고치는 순서는 두 칸입니다. 먼저 **하위 인덱스**로 검색을 타입별로 쪼개고, 그 위에 **하이브리드 RAG**(키워드+시멘틱)를 얹습니다. 여기서 놓치면 안 되는 게, 이 둘은 따로 노는 기술이 아니에요. 하위 인덱스를 쪼갠 타입(works/people/series/concepts/sources)이 그대로 RAG의 컬렉션이 됩니다. ①을 잘 해두면 ③의 정확도가 거기서 따라옵니다.

| Before (Clip 4까지) | After (이 클립) |
|---|---|
| 평면 `index.md` 한 장 — 모든 타입 혼재 | 타입별 `{type}/index.md` + 최상위 라우터 |
| 어떤 질문이든 전체 목록을 1차로 읽음 | 관련 타입 인덱스만 펼쳐서 읽음 |
| `Dune` 영화 vs 시리즈가 비결정적 | 경로 한정 링크 + 타입 컬렉션으로 분리 |
| "씁쓸한 가족 드라마" 같은 의미 질문에 약함 | 시멘틱 검색이 의미로 회수 |

관통하는 한 가지는 **선 긋기**입니다. 어디까지가 설치 0으로 되는 개선이고 어디부터 임베딩·벡터DB가 필요한지를, 얼버무리지 않고 가릅니다. RAG는 적이 아니라 규모로 선을 긋는 도구일 뿐이에요.

이번 클립에서 할 일은 세 단계입니다. 하위 인덱스 만들기 → 개선 확인(전/후 비교) → 하이브리드 RAG 구현. 마지막에 본인 규모로 "여기까지면 충분한가, RAG까지 가야 하나"를 한 줄로 적어 마무리합니다.

## 핵심 개념

### 왜 지금 RAG인가 — 원칙을 어기는 게 아니라 예고한 시점

Part 7 LLM Wiki 챕터 내내 들어온 입장은 "어지간한 규모면 마크다운+검색으로 충분, 정말 커지면 RAG"였습니다. 그럼 지금 RAG를 꺼내는 건 말 바꾸기 아니냐. 아닙니다.

코퍼스가 원본 1,000편에 엔티티 수백~천 단위로 불었어요. 정본 설계서(`~/llm-wiki/30-reports/2026-06-14-rag-index-design.md`)가 미리 정해둔 'RAG 도입 트리거'를 이제 넘은 겁니다. 원칙을 어긴 게 아니라 **예고한 시점에 예고한 대로 도입**하는 거예요. 15편짜리 위키였다면 인덱스를 쪼개든 RAG를 얹든 효과가 거의 안 보였을 겁니다. 1,000편이 깔려 있으니 오늘 개선이 손에 잡히는 거죠.

```text
LLM Wiki 도구 선택 — 규모로 선 긋기
─────────────────────────────────────
수백~수천      → 하위 인덱스 (마크다운, 설치 0)
의미질문·롱테일 → 하이브리드 RAG (임베딩 + 벡터DB)
수만~수십만    → 관리형(Gemini File Search) / Supabase pgvector
```

마지막 줄의 Supabase pgvector는 남의 얘기가 아니에요. 다음 챕터(CH03 유튜브 대시보드)에서 또 만납니다. 오늘 쓰는 벡터DB 원리를 Supabase가 `pgvector`라는 이름으로 똑같이 품고 있거든요. 도구만 다르고 원리는 같습니다.

### 하위 인덱스(MOC) — 평면 index를 타입별로 쪼개기

평면 `index.md` 하나에 모든 타입을 섞어두면 문제가 둘 생깁니다. 하나는 **동명충돌** — `Dune`이라는 이름이 영화에도 시리즈에도 걸쳐 있으면 클로드코드가 어느 쪽 페이지를 읽을지 비결정적이에요. 다른 하나는 **tiered-read 낭비** — 무슨 질문이든 전체 목록을 1차로 훑어야 하니 노이즈가 따라붙습니다.

이걸 타입별 하위 인덱스 + 최상위 라우터로 바꿉니다.

```text
40-wiki/
  index.md           ← 최상위 = 라우터(카탈로그 아님)
                       타입별 하위 인덱스 링크 + 동명충돌 안내표
  works/index.md     ← 작품만
  people/index.md    ← 인물만
  series/index.md    ← 시리즈만
  concepts/index.md  ← 개념만
  sources/index.md   ← (이미 존재) 원본 기사
```

여기서 챙길 게 둘입니다.

첫째, **최상위 `index.md`가 카탈로그가 아니라 라우터**가 됩니다. 전체 페이지를 다 늘어놓는 대신 "작품은 works로, 인물은 people로" 보내는 안내판 역할만 해요. 질문이 들어오면 클로드코드가 라우터를 먼저 읽고 관련 타입의 하위 인덱스만 펼칩니다.

둘째, **동명충돌을 경로 한정 위키링크로 해소**합니다. `[[Dune]]` 대신 `[[works/Dune|Dune]]`(영화)·`[[series/Dune|Dune]]`(시리즈)처럼 경로를 붙여 갈라놓는 거예요. 그리고 라우터에 동명충돌 안내표를 한 칸 둬서 "이 이름은 두 타입에 있다"를 명시합니다. 이 동명충돌 해소가 이번 단계의 진짜 성과예요. 여기까지는 **설치가 0입니다.** 마크다운 파일만 다시 정리하면 끝이에요.

### 하위 인덱스가 곧 RAG의 컬렉션 — 둘은 같은 네임스페이스

이 클립에서 제일 중요한 연결입니다. 타입(works/people/series/concepts/sources)으로 하위 인덱스를 쪼개두면, **그게 그대로 RAG의 컬렉션이자 필터**가 됩니다.

```text
타입 네임스페이스 하나를 둘이 공유
──────────────────────────────────
works/index.md  ─┐
people/index.md ─┤  (Phase 1 하위 인덱스)
series/index.md ─┤
                 │  같은 타입 = 같은 네임스페이스
works 컬렉션  ───┤
people 컬렉션 ───┤  (Phase 3 RAG)
series 컬렉션 ───┘
```

그래서 RAG에서도 `Dune`(works)과 `Dune`(series)이 **컬렉션 분리로 결정적으로** 갈라집니다. 마크다운 단계에서 경로 한정 링크로 풀었던 동명충돌을, 벡터DB 단계에서는 컬렉션 분리로 똑같이 푸는 거죠. 둘은 별개 기술이 아닙니다. "타입으로 쪼갠다"는 같은 발상을 두 층에 한 번씩 적용한 것뿐이에요. ①을 깔아둬야 ③이 그 위에 섭니다.

### 하이브리드 검색 — 왜 키워드와 시멘틱을 둘 다

설명 목록조차 버겁거나 "씁쓸한 가족 드라마 같은 거"처럼 모호한 의미 질문을 잡으려면, 페이지를 임베딩해 시멘틱 검색을 얹어야 합니다. 그런데 시멘틱 하나만으로는 모자라요. 둘을 같이 돌려 점수를 병합하는 이유가 바로 그겁니다.

| 방식 | 강점 | 약점 |
|---|---|---|
| **시멘틱**(의미·임베딩) | "씁쓸한 가족 드라마 같은 거" 모호 질문 | 고유명사·정확한 제목엔 약함 |
| **키워드**(렉시컬·BM25) | "기생충", 음차 표기, 정확한 제목 | 의미 유사엔 약함 |
| **하이브리드**(둘 점수 병합) | 서로의 약점을 메움 | — |

"기생충"이라고 정확히 치면 키워드 검색이 강하고, "씁쓸한 가족 드라마 같은 한국 영화"라고 두루뭉술하게 물으면 시멘틱이 강합니다. 둘을 동시에 돌려 순위를 합치면 서로의 빈틈이 메워져요. 두 검색이 매긴 순위를 점수로 환산해 합산하는 방식인데, 본인이 직접 계산할 일은 없습니다. 클로드코드한테 "키워드랑 시멘틱 같이 돌려서 합쳐줘"라고만 하면 안에서 알아서 처리합니다.

### 용어 30초 정리 (몰라도 됩니다, 궁금하면)

| 말 | 뜻 (우리 식으로) |
|---|---|
| 하위 인덱스(MOC) | 타입별로 쪼갠 목차 페이지 — 사람·에이전트가 먼저 읽는 진입점 |
| 임베딩 | 글을 의미가 담긴 **숫자 벡터**로 바꾼 것 |

## 진행 흐름

세 단계로 진행합니다. Phase 1·2는 마크다운만 만지니 **설치가 0**이고, 임베딩 키와 벡터DB는 Phase 3에서만 필요해요. 키가 없으면 Phase 1·2까지만 해도 이번 클립의 목표는 달성입니다.

### 1. `/part07` 호출 → Clip 5 선택

강의 워크스페이스에서 클로드코드를 켜고 시작합니다.

```text
/part07
```

스킬이 진도를 확인하고 작업 폴더를 준비합니다. LLM Wiki 본체는 강의 워크스페이스 밖 별도 폴더(`~/llm-wiki/`)에 살고, 진도 기록만 `50-my-work` 안에 남는 구조예요. [Clip 3에서 위키 골격을 세울 때](#part-7-03-wiki-design) 정한 그대로입니다.

```text
✓ 50-my-work/Part07-실전/실습36-위키개선/ 진도 폴더 준비
✓ clip-04에서 채운 llm-wiki/ 사용 · 정본 설계서(rag-index-design) 구현
  위키 페이지: NNN장
  원본 기사(raw): 1000편
  설계서: 있음 — 오늘 이걸 구현
```

위키 폴더는 환경에 따라 `40-wiki/` 또는 `wiki/`로 잡힙니다. 둘 다 자동으로 인식하니 신경 안 써도 돼요. 진도 폴더에는 결정 가이드 다이어그램과 RAG 구현 레퍼런스 문서가 같이 복사됩니다.

### 2. STEP 1 — 하위 인덱스 만들기 (설치 0)

명령을 외워 치는 대신, 어떻게 쪼갤지를 말로 푸는 방식으로 시작합니다. 본인이 분할 알고리즘을 짜는 게 아니라 **"타입별로 쪼개고 동명은 경로로 구분해"라고 요청하는 법**을 익히는 단계예요.

복사용 입력입니다.

```text
평면 index.md를 타입별 하위 인덱스로 쪼개줘. works/people/series/concepts
각각 {type}/index.md를 만들고, 각 줄 description은 그 페이지의 summary를
단일 출처로 써. 최상위 index.md는 카탈로그가 아니라 타입별 하위 인덱스로
보내는 라우터로 바꾸고, Dune(영화 works vs 시리즈 series)처럼 같은 이름이
두 타입에 있는 건 동명충돌 안내표로 정리하고 본문 링크는
[[works/Dune|Dune]]·[[series/Dune|Dune]]처럼 경로 한정 위키링크로 바꿔줘.
```

작업이 끝나면 이런 구조가 생깁니다.

```text
✓ works/index.md   생성
✓ people/index.md  생성
✓ series/index.md  생성
✓ concepts/index.md 생성
✓ index.md → 라우터로 전환 (타입 링크 + 동명충돌 안내표)
✓ Dune → [[works/Dune]] / [[series/Dune]] 경로 한정 링크로 분리
```

확인할 건 두 가지예요. `{type}/index.md`가 타입별로 생겼는지, 그리고 최상위 `index.md`가 전체 카탈로그가 아니라 라우터로 바뀌었는지. `Dune` 같은 동명이 타입별로 갈라져 있으면 성공입니다.

여기서 하나 챙겨둘 게 있어요. 각 줄의 description을 **그 페이지의 summary를 단일 출처로** 쓰라고 한 데는 이유가 있습니다. 이 description이 나중에 Phase 3 RAG의 검색 품질을 좌우하거든요. summary를 본문 기반으로 충실히 써두면 시멘틱 검색이 잘 잡히고, 대충 써두면 RAG도 따라서 흐려집니다. ①에서 description을 제대로 써둔 게 ③의 정확도로 돌아온다는 말이 여기서 나와요.

### 3. STEP 2 — 개선 확인 (전/후 비교, 설치 0)

이제 전체를 읽지 않고 라우터 → 해당 타입 인덱스만 펼쳐서 답하게 해봅니다. 전(평면 전체 읽기)과 후(타입 인덱스만)의 차이를 직접 눈으로 보는 게 목적이에요.

복사용 입력입니다.

```text
"봉준호 영화의 특징"을 답하는데, 최상위 index(라우터)를 먼저 읽고 관련
타입의 하위 인덱스만 펼쳐서 후보를 고른 다음 그 페이지들만 읽고 답해줘.
어떤 인덱스를 읽었는지랑 출처도 같이.
```

"봉준호"는 인물 질문이라 클로드코드가 `people/index.md`(+필요시 works)만 펼칩니다. 작품·시리즈 전체는 건드리지 않아요. 빠르고, 노이즈가 적고, 동명충돌도 안 생깁니다.

| | 전 (평면 전체 읽기) | 후 (타입 인덱스만) |
|---|---|---|
| 읽는 양 | 전체 목록 | 관련 타입만 |
| 속도 | 느림 | 빠름 |
| 동명충돌 | 있음 | 없음 (타입 분리) |
| 설치 | 0 | 0 |

여기까지가 **설치 0으로 되는 개선**입니다. 본인 위키가 어지간한 규모라면 여기서 멈춰도 괜찮아요. "느려지거나 빠지는 게 체감 안 된다"면 RAG는 아직 필요 없습니다. 그게 다음 단계로 갈지 말지를 가르는 기준이에요.

### 4. STEP 3 — 하이브리드 RAG 구현 (임베딩 키 필요)

설명 목록조차 버겁거나 의미 질문을 잡아야 하면 RAG로 넘어갑니다. 페이지를 임베딩해 시멘틱 검색을 얹고, 동시에 키워드(BM25) 검색을 돌려 점수를 병합해요. (설계서 Part B)

시멘틱 검색에는 **Gemini API 키**(`GEMINI_API_KEY`)가 필요합니다. Clip 4나 Part 6에서 이미 발급했으면 그대로 재사용하면 되고, 없으면 발급부터 합니다.

```text
Gemini API 키 발급 (시멘틱 검색에만 필요)
─────────────────────────────────────
1. https://aistudio.google.com/apikey 접속
2. "Create API key" → 키 복사
3. .rag/.env 에 한 줄: GEMINI_API_KEY=...
   (키 파일은 git 무시 — 커밋 X)
```

**① 인덱스 구축** — 복사용 입력입니다.

```text
.rag/ 폴더에 RAG 인덱스를 만들어줘. 위키 페이지와 원본 소스를 2-tier
(요약=summary 한 청크, 본문=섹션별 청크)로 쪼개고, 타입(works/people/series/
concepts/sources)별 컬렉션으로 나눠서 Gemini gemini-embedding-001로 임베딩해
(색인은 RETRIEVAL_DOCUMENT). 청크마다 page·type·title·year·tier·sources
메타데이터를 붙여줘. 저장은 로컬 벡터DB로.
```

이 한 줄 요청 안에 RAG 구현 사양이 거의 다 담겨 있어요. 표로 풀어보면 이렇습니다.

| 항목 | 값 |
|---|---|
| 인덱싱 단위 | **2-tier 청킹** — 요약 tier(페이지 summary 1청크) + 본문 tier(섹션별 1청크). 원본 소스 1,000편도 같이 인덱싱(롱테일 회수) |
| 컬렉션 분리 | 타입별(works/people/series/concepts/sources) — Phase 1 하위 인덱스와 같은 네임스페이스 |
| 임베딩 모델 | **`gemini-embedding-001`** (멀티링구얼 — 한/영 혼재 필수). 색인 `RETRIEVAL_DOCUMENT` / 쿼리 `RETRIEVAL_QUERY` task_type 구분 |
| 저장소 | 로컬 경량 — ChromaDB / sqlite-vec (수천 벡터 규모). 위키 옆 `.rag/`에 저장, git 무시 |

왜 통째로 안 하고 2-tier로 쪼개느냐. 글을 통째로 숫자화하면 "대충 무슨 얘기"까지만 잡히기 때문이에요. 요약 tier(페이지 summary 1청크)는 빠르게 후보를 추리는 용도, 본문 tier(섹션별 1청크)는 정밀 회수·인용용입니다. 위키 페이지가 없는 롱테일 엔티티(조연 같은)도 원본 기사를 같이 인덱싱해두면 기사 내용으로 검색돼요.

**② 하이브리드 검색 적용** — 복사용 입력입니다.

```text
이제 질문을 받으면 키워드(BM25) 검색과 시멘틱(임베딩) 검색을 동시에 돌려서
점수를 병합하고, type/tier로 필터한 다음 상위 후보 페이지만 읽어서 답하게
해줘. 쿼리는 RETRIEVAL_QUERY로 임베딩하고, 타입 컬렉션에서 못 찾으면
sources로 폴백해. "씁쓸한 가족 드라마 같은 한국 영화"로 한번 검색해봐.
```

확인할 건 이겁니다. 키워드로는 안 잡히는 의미 질문이 시멘틱으로 회수되고, 정확한 제목·고유명사는 키워드로 보강됩니다. 둘을 병합한 결과로 답이 나오고 출처가 인용되면 성공이에요. 타입 컬렉션에서 결과가 빈약하면 `sources` 컬렉션으로 폴백해서, 자체 페이지가 없는 엔티티도 원본 기사로 회수합니다.

쿼리가 들어왔을 때 안에서 도는 순서는 이렇습니다. 외울 건 없고, 한 번 훑어두면 흐름이 잡혀요.

```text
질문
 → ① alias 정규화 + 의도/타입 추론  ("Parasite"→["Parasite","기생충"])
 → ② 하이브리드 검색 (시멘틱 + BM25 + type/tier 필터)
 → ③ 리랭크 (상위 k)
 → ④ 후보 페이지만 펼쳐 읽고 [[링크]] 1홉 보강
 → ⑤ 근거로만 답 + 정본 인용 + provenance
```

여기서 **RAG가 바꾸는 건 '후보 찾기'(②)뿐**이에요. 읽고·인용하고·근거 대는 원칙은 Clip 3·4에서 세운 그대로입니다. 검색만 똑똑해질 뿐, 답하는 방식은 그대로예요.

> **정직 가드** — 하이브리드도 만능은 아닙니다. 결과는 세 가지에 좌우돼요. 가져오는 개수(top-K), 임베딩 모델 품질(한국어면 멀티링구얼 필수), 그리고 가장 결정적인 **summary 품질**. Phase 1에서 description(summary)을 본문 기반으로 충실히 써둔 게 여기서 정확도로 돌아옵니다. "토큰 N배 절감·recall 몇 %" 같은 정량 단정은 하지 마세요. 그런 수치는 출처가 약합니다. 대신 **전/후를 본인 볼트로 직접 측정**해서 체감으로 판단하는 게 정직한 방법이에요.

### 5. 경계 정리 — 내 규모는 어디?

마지막으로 세 단계의 경계를 한 번에 정리합니다.

| | 평면 전체 읽기 | 하위 인덱스 | 하이브리드 RAG |
|---|---|---|---|
| 속도 | 느림 | 빠름 | 빠름 |
| 동명충돌 | 있음 | 해소 | 해소(타입 컬렉션) |
| 의미 질문 | 약함 | 약함 | 강함 |
| 롱테일 회수 | 누락 | 페이지 있어야 | 소스 청크로 회수 |
| 설치 | 0 | 0 | 임베딩·벡터DB |

선 긋기는 이렇게 잡아두면 됩니다. 하위 인덱스(Phase 1·2)는 설치 0으로 동명충돌과 tiered-read 낭비를 해소하니, 어지간한 규모는 여기서 충분. 하이브리드 RAG(Phase 3)는 의미 질문·롱테일 회수가 필요하거나 설명 목록조차 버거운 규모. 더 커지면 관리형(Gemini File Search) 또는 Supabase pgvector(CH03에서 또 만남)로 넘어갑니다.
