# Part 7 / Clip 4: 위키 짓기 — 자료 투입부터 질문까지

> 강의 영상: Part 7 / Ch 02 / Clip 4 (~25분)
> 만드는 것: 영화 자료로 채운 동작하는 위키 (`wiki/` 인물·작품 페이지 + frontmatter `description`·`sources` + `[[링크]]`) + 질문 1건 + 그 답을 다시 쌓은 페이지
> 시연 주제: 영화 감상 위키 (봉준호·송강호·기생충·괴물 …)

---

## 이 클립에서 만드는 것

[지난 클립](#part-7-03-wiki-design)에서 위키 설계도 4칸을 그렸어요. 무슨 자료를 모을지, `raw`에 무엇을 넣을지, `wiki` 페이지를 어떤 엔티티 단위로 쪼갤지, 정리 규칙(`description`·`sources`·이름·`[[링크]]`)은 무엇으로 둘지. 여기까지는 말로만 정한 거였죠. 오늘은 그 설계도를 들고 **실제로 손을 댑니다.** 빈 위키 뼈대에 자료를 넣고, AI가 읽어서 정리하고 연결하고, 거기에 질문까지 한 바퀴 돌려봐요.

손에 익히는 자료는 **영화 감상 위키**입니다. 봉준호, 송강호, 기생충, 괴물처럼 누구나 아는 도메인이라 정리 결과를 바로 알아볼 수 있어요. 본인 주제(마케팅 사례나 분야 자료)는 **똑같은 방식**으로 나중에 적용하면 됩니다. 오늘은 도메인이 익숙한 영화로 손에 박아두는 날이에요.

오늘 도는 건 [clip-03에서 본 4동사 루프](#part-7-03-wiki-design) 중 앞 세 개입니다.

```text
투입(ingest)  →  정리(compile)  →  질문(query)   [+ (선택) 그래프]
raw에 자료 넣기   raw 읽어 wiki    위키에 묻고
(아직 정리 X)     페이지로 정리·연결  좋은 답은 다시 저장
```

네 번째 동사인 **점검(lint)**은 위키가 천 편 규모로 커졌을 때 깨지는 곳을 고치는 단계입니다. 그건 [다음 클립](#part-7-05-wiki-improve)에서 다뤄요. 오늘은 위키가 "생기고 자라는" 장면까지입니다.

| Before | After |
|---|---|
| 위키 뼈대만 있고 자료가 0 (clip-03 설계도) | 인물·작품 페이지가 채워진 동작하는 위키 |
| AI에게 매번 같은 자료를 다시 설명한다 | 한 번 정리한 페이지에서 답이 나온다 |
| 질문하면 답하고 끝 (검색) | 좋은 답이 새 페이지로 쌓인다 (복리) |

이번 클립에서 할 일은 세 단계(+선택 1)입니다. 투입 → 정리 → 질문, 그리고 원하면 그래프까지. 영상에서 한 줄 입력하면 본인도 같은 줄을 그대로 따라치고 결과를 같이 확인하는 **따라치기** 클립이에요.

## 핵심 개념

### 라이브 15편 + 대량 공개 1,000편 — 둘을 섞는 이유

오늘 위키에 들어가는 영화 자료는 두 덩어리예요. 직접 정리할 **15편**, 그리고 미리 빌드해둔 **1,000편**입니다.

| | 무엇 | 누가 |
|---|---|---|
| **라이브** | 영화 raw 15편을 직접 정리시켜 위키가 생기는 걸 따라치기 | 본인이 손으로 |
| **대량 공개** | raw 1,000편을 미리 빌드해둔 영화 위키를 열어 "이만큼 쌓이면 이렇게 된다" | 미리 빌드된 완성본 |

천 편을 라이브로 정리할 순 없어요. 그러면 영상 한 클립으로 안 끝나니까요. 그래서 **소량은 직접 만들어 원리를 손에 익히고, 대량은 미리 빌드한 완성본으로 결과를 본다**, 이렇게 두 갈래로 갑니다. 둘 다 `raw/`에서 빌드한 결과라 본질은 같아요. 라이브 15편은 서로 연결되도록 묶음으로 골랐습니다(봉준호 작품군, 박찬욱 작품군처럼). 그래서 적은 양으로도 `[[링크]]`가 엮이는 걸 바로 볼 수 있어요.

### 정리(compile)의 정체 — 원본을 베껴 정리 페이지를 새로 쓴다

가장 중요한 동사가 **정리(compile)**예요. 한 문장으로 말하면, AI가 `raw/`의 거친 원본을 읽고 `wiki/`에 **인물·작품 단위의 정리 페이지를 새로 쓰는** 일입니다.

여기서 [clip-03의 가장 중요한 약속](#part-7-03-wiki-design)을 다시 떠올리세요. **`raw/`의 원본은 AI가 절대 고치지 않습니다.** 원본은 박물관 유리장 안에 있고, AI는 그걸 보고 베껴서 `wiki/`에 새 페이지를 만들 뿐이에요. 그러니까 정리는 "원본 수정"이 아니라 "원본을 재료로 새 페이지를 만드는" 일입니다.

```text
raw/0001-parasite-review.md   (원본 감상문 — 안 고침)
raw/0026-parasite-review.md   (또 다른 기생충 감상문 — 안 고침)
raw/0002-...-murder-review.md (살인의 추억 감상문 — 안 고침)
          │ AI가 읽고 베껴서
          ▼
wiki/people/봉준호.md   ← 여러 감상문에 흩어진 봉준호 이야기가 한 페이지로
wiki/works/기생충.md    ← 여러 감상문의 기생충 정보가 한 페이지로
wiki/works/괴물.md
```

여기서 중요한 건 **엔티티 단위로 모인다**는 거예요. 감상문 하나하나를 그대로 페이지로 만드는 게 아닙니다. 거기 등장하는 인물(봉준호·송강호)과 작품(기생충·괴물)을 페이지 단위로 잡아요. 그러면 여러 raw 파일에 흩어진 같은 인물 이야기가 한 페이지로 합쳐집니다. 이게 검색과 다른 점이에요. 검색은 질문할 때마다 조각을 새로 긁지만, 위키는 정리할 때 한 번 합쳐둡니다.

### 페이지마다 description·sources를 frontmatter에 — 왜 처음부터 박나

[clip-03에서 정한 규칙](#part-7-03-wiki-design)이 정리할 때마다 똑같이 적용됩니다. 각 페이지 맨 위에 frontmatter로 `description`(본문 전체를 바탕으로 한 줄 요약)과 `sources`(출처)를 넣는 규칙이에요. 결과 페이지는 이런 모습입니다.

```text
wiki/people/봉준호.md
─────────────────────
---
description: 한국 영화감독. 기생충·괴물·살인의 추억 등에서 계급·가족·
  공간 연출을 반복 변주하며 장르를 뒤섞는다. 봉준호, 봉테일, 송강호
  협업, 칸 황금종려상, 블랙 코미디 같은 키워드로 검색됨.
sources:
  - raw/0001-parasite-review.md
  - raw/0002-memories-of-murder-review.md
  - raw/0003-the-host-review.md
tags: [감독, 한국영화]
---

# 봉준호

[[기생충]], [[괴물]], [[살인의 추억]]을 만든 감독. …
본문에서 [[송강호]]와의 협업, [[블랙 코미디]] 톤 전환을 …
```

`description`을 **처음부터** 규칙으로 박아두는 데는 이유가 있어요. 위키가 커지면 클로드코드가 모든 페이지를 다 읽지는 않습니다. 목차(`index.md`)의 한 줄 설명만 훑어보고, 정말 필요한 페이지만 펼쳐서 읽어요. 이걸 계층적 읽기(tiered read)라고 부르는데, [다음 클립](#part-7-05-wiki-improve)에서 제대로 다룹니다. 지금은 한 가지만 기억하면 돼요. **검색 품질이 곧 description 품질**이라는 것. 그래서 description은 추상적인 한 줄이 아니라, 그 페이지에 무슨 내용과 키워드가 있는지 알 수 있는 한 줄이어야 합니다. 정리시킨 직후, 생성된 description이 한두 문장 이상으로 충분한지 한 번 훑어보세요.

`sources`는 "이 정리, 어디서 나온 거지?" 싶을 때 원본으로 돌아갈 다리예요. 정량적 평가나 별점이 아니라 **출처 경로**를 남기는 겁니다. 출처가 붙어 있어야 위키를 믿고 쓸 수 있죠. 별도 `sources.md` 파일을 만들지 말고, **각 페이지 frontmatter 안에** 두세요. 이게 규칙입니다.

### 같은 대상은 같은 이름 — [[링크]]가 엮이려면

`[[링크]]`로 페이지를 엮으려면 약속이 하나 필요해요. **같은 대상은 항상 같은 이름으로 부른다**는 겁니다. 한 감상문엔 "봉준호 감독", 다른 감상문엔 "봉 감독"이라고 쓰여 있어도, 위키 페이지 이름과 링크는 `[[봉준호]]` 하나로 통일해야 해요. 그래야 흩어진 언급이 한 페이지로 모이고 링크가 끊기지 않습니다.

영화는 표기가 갈리기 쉬운 도메인이라 규칙을 하나 더 둡니다. **한국 작품은 한글 정본, 외국 작품은 원제**예요. `기생충`은 한글로, `Dune`은 원제로. "듄"·"Dune (2021)"·"듄: 파트2" 같은 변형은 정본 한 곳으로 모아 중복 페이지가 갈라지지 않게 합니다. 이런 표기 통일과 중복 제거(같은 이름인데 타입이 다른 동명충돌까지)를 제대로 손보는 건 [다음 클립](#part-7-05-wiki-improve)이에요. 오늘은 "같은 대상은 같은 이름으로 통일해서 링크 걸어줘" 한 줄이면 충분합니다.

링크에는 원칙이 하나 더 있어요. **`[[X]]`는 대상 페이지가 실제로 있을 때만 겁니다.** 페이지가 없는 단역 배우까지 전부 링크로 만들면 끊긴 링크(죽은 링크)만 늘어나요. 페이지가 없는 대상은 그냥 본문 텍스트로 두세요. 나중에 그 대상이 두 번 이상 등장하거나 질문 대상이 되면, 그때 페이지로 승격하면 됩니다.

### 질문(query)의 답이 다시 쌓인다 — 복리

오늘 클립에서 꼭 눈으로 봐야 할 장면이 **질문의 답이 다시 위키에 쌓이는** 순간입니다. LLM Wiki를 검색과 가르는 결정적 차이가 여기예요.

```text
검색(RAG)            : 질문 → 조각 긁어 답 → 끝 (답은 흩어짐)
LLM Wiki            : 질문 → 위키 페이지 근거로 답 → 그 답을 새 페이지로 저장
                                                  └→ 다음 질문의 재료가 됨
```

"봉준호 영화의 공간 연출"을 위키에 물으면, AI는 봉준호·기생충·괴물 페이지를 근거로 답을 합성하고 출처를 같이 줍니다. 검색이라면 여기서 끝이에요. 그런데 위키에서는 그 좋은 답을 **`wiki/queries/`에 새 정리 페이지로 저장**하고, 관련 인물·작품 페이지에 `[[링크]]`로 연결합니다. 그러면 탐색한 결과조차 자산이 돼요. 다음에 비슷한 질문을 하면 이미 정리된 그 페이지에서 답이 나옵니다. 쓸수록 위키가 똑똑해지는 복리가 바로 이 장면에서 시작돼요.

## 진행 흐름

> **입력 두 단 짚기**: 아래에는 두 종류의 입력이 섞여 있어요. **도입 멘트**는 평소 대화처럼 "~하려는데 어떻게?"로 묻는 줄이고, **실행 입력**은 AI에게 실제로 동작을 시키는 줄이라 "~해줘" 형태입니다. 도입 멘트로 방법을 먼저 묻고, AI가 알려준 방법을 실행 입력으로 따라치는 흐름이에요. 이 2단은 영상에서 말로 짚어줄 거예요. [Part 6 내내 익힌 그 패턴](#part-6-01-skill-intro) 그대로입니다.

### 1. `/part07` 호출 → Clip 4 선택

강의 워크스페이스에서 클로드코드를 켜고 시작합니다.

```text
/part07
```

스킬이 진도를 확인하고 클립 선택지를 띄웁니다. Clip 4를 선택하면 자동 셋업이 두 가지를 준비해요. clip-03에서 만든 빈 위키(`~/llm-wiki/`)의 `raw/`에 **라이브용 영화 자료 15편**을 넣고, 진도 폴더를 만듭니다.

```text
✓ ~/llm-wiki/raw/에 영화 자료 15편 투입 완료 (정리는 아직)
  전체 1,000편 raw는 시드에 그대로 있음 (대량 빌드·공개용)
✓ ~/fastcampus-cc/50-my-work/Part07-실전/실습35-위키짓기/ 진도 폴더 준비
```

여기서 두 폴더가 분리돼 있다는 걸 다시 확인하세요. **위키 본체는 `~/llm-wiki/`**(별도 워크스페이스)에 살고, **진도 기록(README)만 `50-my-work/`**에 남습니다. [clip-03에서 정한 규칙](#part-7-03-wiki-design) 그대로예요. 위키는 강의가 끝나도 계속 자라는 본인 자산이라, 강의 폴더 안에 가두지 않습니다.

clip-03을 안 거쳐서 빈 위키가 없어도 걱정 마세요. 자동 셋업이 최소 골격(`raw/`·`wiki/`)을 보강해줍니다. 다만 정리 규칙이 담긴 `CLAUDE.md`는, clip-03에서 만든 게 있으면 그대로 쓰는 게 좋아요.

### 2. STEP 1 — 투입(ingest): 영화 자료 훑어보기

자료는 셋업이 이미 `raw/`에 넣어줬어요. 첫 단계는 "정리 말고 훑어만" 보는 겁니다. 무슨 인물과 작품이 반복 등장하는지, 지형부터 파악하는 거예요.

**실행 입력:**
```text
llm-wiki/raw/ 폴더의 영화 자료들을 읽고, 어떤 주제(인물·작품·장르)가
반복되는지 먼저 훑어줘. 정리는 아직 하지 마.
```

클로드코드가 raw 파일들을 읽고 반복 등장하는 대상을 목록으로 보고합니다. 이런 모습이에요.

```text
반복 등장 인물: 봉준호(여러 편), 송강호, 박찬욱, 류승룡 …
반복 등장 작품: 기생충, 살인의 추억, 괴물, 설국열차, 올드보이 …
장르: 블랙 코미디, 스릴러, SF …
(wiki/는 아직 비어 있음 — 정리 전이라 정상)
```

| 확인 | AI가 반복 인물·작품을 목록으로 보고 / `wiki/`는 아직 비어 있음 |

이 단계에서 `wiki/`가 비어 있는 건 정상이에요. 투입은 자료를 넣고 지형을 보는 데까지고, 정리 페이지는 다음 STEP에서 만듭니다.

### 3. STEP 2 — 정리(compile): 위키 페이지 만들기 ★

오늘의 핵심 단계입니다. 먼저 **도입 멘트**로 방법을 묻고, 그다음 **실행 입력**으로 정리를 시킵니다.

**도입 멘트 ("~하려는데 어떻게?"):**
```text
raw에 있는 영화 자료를 인물·작품별 위키 페이지로 정리하고 싶은데,
서로 연결도 하고 나중에 검색도 잘 되게 하려면 어떻게 하면 좋을까?
```

클로드코드가 방법을 제안하면(엔티티 단위로 쪼개기, frontmatter에 요약·출처, 같은 이름 통일, 링크 연결 등), 그 방법 그대로 실행 입력을 던집니다. 사실 이 규칙들은 [clip-03에서 `CLAUDE.md`에 박아둔 것](#part-7-03-wiki-design)이라 매번 똑같이 적용돼요.

**실행 입력 (라이브):**
```text
raw 영화 자료를 읽고 wiki/에 주제별 페이지(인물·작품)를 만들어줘.
각 페이지 맨 위에 frontmatter로 description(본문 전체를 바탕으로 한 줄
요약)과 sources(출처)를 넣고, 관련 페이지끼리 [[ ]] 링크를 걸어줘.
같은 대상은 같은 이름으로 통일해줘.
```

클로드코드가 raw를 읽어 `wiki/`에 인물·작품 페이지를 만듭니다. 결과는 이런 모습이에요.

```text
wiki/
├── people/
│   ├── 봉준호.md      (description·sources / [[기생충]] [[괴물]] 링크)
│   ├── 송강호.md
│   └── 박찬욱.md
├── works/
│   ├── 기생충.md
│   ├── 괴물.md
│   ├── 살인의 추억.md
│   └── 설국열차.md
├── concepts/
│   └── 블랙 코미디.md
└── index.md          ← 목차(MOC). 모든 페이지 + 한 줄 description
```

| 확인 | `wiki/`에 인물·작품 페이지 생성 / 각 페이지 frontmatter에 `description:`·`sources:` / 본문에 `[[링크]]` |

**description 품질 확인**: 생성된 페이지를 한두 장 열어서, description이 한두 문장 이상으로 충분한지 훑어보세요. 너무 짧으면(예: "봉준호 페이지") [다음 클립의 "설명으로 골라 읽기"](#part-7-05-wiki-improve)가 부정확해집니다. 부실하다 싶으면 "각 페이지 description을 본문 키워드까지 담은 한두 문장으로 다시 써줘" 한 줄이면 됩니다.

여기서 챙길 게 하나 더 있어요. 정리가 끝나면 클로드코드가 `index.md`(목차)와 `log.md`(작업 기록)도 갱신합니다. `index.md`는 클로드코드가 질문할 때 **가장 먼저 읽는 입구**(목차)고, `log.md`는 "언제 무엇을 정리했나"를 한 줄씩 덧붙이기만 하는 타임라인이에요. 자료가 많으면 한 번에 다 시키지 말고 **100편씩 배치**로 나눠 정리하는 게 안전합니다. 이 배치 패턴도 다음 클립에서 더 다뤄요.

**대량 공개:** 여기서 영상은 raw 1,000편을 미리 빌드해둔 영화 위키를 엽니다.

> "소량은 방금 직접 만들었고, 1,000편을 다 정리하면 이렇게 인물·작품이 촘촘히 엮인 위키가 됩니다. 다만 이 정도로 커지면 '전체 읽기'가 삐걱대는 곳도 생기는데, 그건 다음 클립에서 솔직하게 봅니다."

### 4. STEP 3 — 질문(query): 답을 다시 쌓기 ★

이제 위키에 질문합니다. 위키 페이지를 근거로 답하고 출처까지 같이 달라고 시켜요.

**실행 입력 (질문):**
```text
이 위키에서 "봉준호 영화의 공간 연출"을 정리해줘.
근거가 된 페이지를 출처로 같이 알려줘.
```

클로드코드가 `index.md`에서 관련 페이지를 찾고(봉준호·기생충·괴물 …), 그 페이지들을 근거로 답을 합성합니다. 답 끝에는 어느 페이지를 근거로 썼는지 출처가 붙어요.

```text
봉준호 영화의 공간 연출 — 핵심 정리

- 계급을 대사가 아니라 공간의 높낮이로 체감시킴
  (반지하 / 거실 / 지하실 → [[기생충]])
- 폭우·계단·냄새 같은 반복 모티프로 공간에 감정을 입힘
- 한정된 공간(열차 칸)을 사회 구조로 치환 → [[설국열차]]

근거 페이지: [[봉준호]], [[기생충]], [[설국열차]], [[괴물]]
```

| 확인 | 위키 페이지를 근거로 답 + 출처 |

검색이라면 여기서 끝이에요. 그런데 위키에서는 한 단계가 더 있습니다. **이 좋은 답을 위키에 다시 저장**하는 거예요.

**실행 입력 (재저장 — 복리):**
```text
방금 답을 wiki/queries/에 새 페이지로 저장하고(frontmatter description·
출처 포함), 관련 인물·작품 페이지에 [[링크]] 걸어줘.
```

클로드코드가 답을 새 페이지로 저장하고 링크를 연결합니다.

```text
✓ wiki/queries/2026-06-14-봉준호-공간연출.md 생성
  (frontmatter description·sources + 본문 [[링크]])
✓ [[봉준호]] · [[기생충]] · [[설국열차]] 페이지에서 이 페이지로 역링크
✓ index.md · log.md 갱신
```

| 확인 | `wiki/`에 새 정리 페이지 추가 + 링크 연결. 탐색이 자산으로 쌓임 |

이 장면이 [clip-03에서 "말로만 들었던" 복리](#part-7-03-wiki-design)를 눈으로 확인하는 순간이에요. 질문해서 나온 답이 사라지지 않고 페이지로 남아서, 다음 질문의 재료가 됩니다.

### 5. (선택) STEP 4 — 그래프 그려보기

시간이 남으면 `[[링크]]` 관계를 그림으로 봅니다. 옵시디언 같은 별도 앱 없이도 클로드코드가 직접 그려줘요.

**실행 입력:**
```text
wiki/의 [[ ]] 링크 관계를 점·선 그래프 HTML 한 파일로 만들어줘.
브라우저에서 드래그·줌 되게.
```

HTML 파일 하나가 생기고, 브라우저로 열면 인물·작품이 점으로, `[[링크]]`가 선으로 그려진 그래프가 나옵니다. 봉준호 점에서 기생충·괴물·설국열차로 선이 뻗어 있는 모습이 눈에 들어와요.

> 이 그래프는 **시각화 전용**이에요. 위키의 검색은 description 기반으로 돌아가지, 이 그래프를 따라가며 답을 찾는 게 아닙니다(graph 검색은 우리 강의 범위 밖). 그래프는 "내 위키가 얼마나 엮였나"를 눈으로 확인하는 용도입니다.

## 결과물

이번 클립이 끝나면 위키 본체와 진도 기록 두 곳에 남습니다.

| 결과물 | 위치 | 설명 |
|---|---|---|
| 인물·작품 정리 페이지 | `~/llm-wiki/wiki/people/`, `~/llm-wiki/wiki/works/` | 각 페이지 frontmatter에 `description`·`sources` + 본문 `[[링크]]` |
| 목차·로그 | `~/llm-wiki/wiki/index.md`, `~/llm-wiki/log.md` | 질문 때 먼저 읽는 입구(MOC) + 작업 타임라인 |
| 저장된 질문 답 | `~/llm-wiki/wiki/queries/{날짜}-{주제}.md` | 질문 답을 다시 쌓은 새 페이지 (복리) |
| (선택) 그래프 | `~/llm-wiki/` 안 HTML 한 파일 | `[[링크]]` 관계 시각화 (검색 아님) |
| 진도 README | `~/fastcampus-cc/50-my-work/Part07-실전/실습35-위키짓기/README.md` | 오늘 한 것·쌓인 것·회고 (자동 작성) |

`완료` 또는 `/wrap`을 입력하면 스킬이 정리합니다. 먼저 결과물을 검증해요. `raw` 편수, `wiki` 페이지 수, `description`이 달린 페이지 수를 세어 보고합니다. `wiki` 페이지가 0이면 정리(STEP 2)에서 멈춘 거고, `description`이 0이면 규칙이 안 먹은 거라 다시 시키도록 안내해줘요.

그다음 `실습35-위키짓기/README.md`가 자동으로 작성되고, `progress.json`의 `practice_completed`에 `"실습 35"`가, `projects_completed`에 `"llm-wiki"`가 추가됩니다. 마지막에 한 줄 회고를 자유 입력으로 받아 README에 기록해요. "검색과 다르게 '쌓인' 것은 무엇이었나"를 본인 말로 적으면 됩니다.

[Part 7의 결과물 저장 규칙](#part-7-03-wiki-design)을 다시 확인해보면, 위키 본체(`~/llm-wiki/`)와 진도 기록(`50-my-work/`)이 분리돼 있어요. 한쪽은 계속 자라는 자산, 한쪽은 강의 진도 기록입니다.

## 자주 막히는 포인트

| 증상 | 원인 | 해결 |
|---|---|---|
| 한 번에 다 시켰더니 멈춤 | 자료를 통째로 한 번에 정리 | "100편씩 나눠서 정리해줘" — 배치로 끊어 시키기 (배치 패턴은 clip-05에서 더) |
| `description`이 안 붙음 | 정리 규칙이 안 먹음 | "각 페이지 맨 위에 본문 기반 한 줄 description을 꼭 넣어줘"로 다시 요청 |
| `sources`(출처)가 안 붙음 | frontmatter 누락 | "frontmatter sources에 출처(raw 파일 경로)를 꼭 남기라고 다시 시켜줘" |
| `description`이 한 단어로만 나옴 | 본문 기반이 아니라 제목만 옮김 | "본문 키워드까지 담은 한두 문장으로 다시 써줘" — 검색 품질 = description 품질 |
| 링크가 안 걸림 | 같은 대상을 다른 이름으로 부름 | "같은 대상은 같은 이름으로 통일해서 링크 걸어줘" (표기 통일·중복 제거는 clip-05) |
| 끊긴 링크(죽은 링크)가 생김 | 없는 페이지까지 `[[ ]]`로 만듦 | "대상 페이지가 실제 있을 때만 링크하고, 없으면 텍스트로 둬" |
| `raw`가 비어 있음 | 시드 복사 실패 | "seed-movie-wiki/raw 시드가 `~/llm-wiki/raw/`에 복사됐는지 확인해줘" |
| `wiki`가 비어 있는데 정상인가요? | STEP 1(투입) 직후 | 정상입니다. 정리(STEP 2) 전이라 비어 있어요. STEP 2에서 채워집니다 |
| 질문 답이 저장이 안 됨 | 재저장 입력을 안 줌 | 질문 답을 받은 뒤 "방금 답을 wiki/queries/에 새 페이지로 저장해줘"를 별도로 |
| 그래프 보려면 옵시디언 깔아야 하나요? | 위키 앱이 필요하다는 오해 | 아니요. "그래프 그려줘"라고 하면 클로드코드가 HTML로 그려줍니다 |
| 영화 말고 본인 주제로 해보고 싶음 | 응용 욕구 | OK. 단 오늘은 영화로 손에 익히고, 본인 주제는 같은 방식으로 나중에 적용 |
| 위키를 강의 폴더 안에 만들어야 하나? | 저장 위치 혼동 | 위키 본체는 별도(`~/llm-wiki/`), 진도 기록(README)만 `50-my-work/`에 |

자유 진행 중 막히면 `막혔어요` / `도와줘`로 클로드코드에 도움을 요청할 수 있어요. 회고도 자유 입력입니다. 정해진 보기 중에 고르는 게 아니라 본인 말로 적으면 됩니다.
