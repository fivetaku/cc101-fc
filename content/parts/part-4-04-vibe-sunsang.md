# Part 4 / Clip 4: vibe-sunsang — 내 AI 활용 점검받기

> 강의 영상: Part 4 / Ch 01 / Clip 4 (~17분)
> 만드는 것: 본인 사용 패턴 6축 진단 리포트 + Part 5 시작 전 보완 영역 1~2개

---

## 이 클립에서 만드는 것

Part 2부터 여기까지 오면서 클로드코드와 꽤 많은 대화를 나눴을 겁니다. 그 대화는 자동으로 `~/.claude/projects/`에 JSONL 형식으로 저장됩니다. 평소에는 거의 볼 일이 없는 폴더지만, 여기에 내가 클로드코드를 어떻게 써왔는지 기록이 남아 있습니다.

vibe-sunsang(바이브 선생님)은 그 기록을 읽고 사용 패턴을 점검합니다. 작업 분해, 검증, 오케스트레이션, 실패 복구, 컨텍스트 제공, 메타 인지까지 6개 축으로 점수를 매기고 강점과 약점을 짚어줍니다. Part 5에 들어가기 전에 보완하면 좋은 영역도 1~2개로 추립니다.

이 클립은 Part 4의 마지막입니다. 단순히 “잘 썼는지” 확인하는 시간이 아닙니다. 여기서 받은 진단이 Part 5에서 어느 클립을 더 신경 써야 할지 알려주는 기준이 됩니다.

| Before | After |
|---|---|
| 내가 클로드코드를 잘 쓰고 있는지 감으로만 안다 | 6축 점수로 강점과 약점이 보인다 |
| Part 5에서 뭘 더 봐야 할지 막연하다 | 보완 영역이 Part 5 클립 가이드가 된다 |
| 같은 실수를 반복한다 | 안티패턴 1~3개를 알고 시작한다 |

시연은 시작 → 변환 → 멘토링 순서로 진행합니다. 진단 결과가 강사와 다르게 나와도 괜찮습니다. 본인 사용 기록을 기준으로 보기 때문에 당연히 다르게 나옵니다.

## 핵심 개념

### 클로드코드는 모든 대화를 자동으로 저장합니다

클로드코드는 별도 설정 없이도 세션 대화를 `~/.claude/projects/` 안에 JSONL 파일로 저장합니다. 프로젝트별로 폴더가 나뉘고, 세션마다 파일 하나가 쌓입니다. vibe-sunsang은 이 파일들을 읽어서 분석합니다.

새로 기록을 만드는 게 아닙니다. 이미 쌓여 있는 기록을 분석에 쓰는 방식입니다. 그래서 Part 2부터 진행한 작업이 그대로 분석 대상이 됩니다.

### 6축은 “무엇을 만들었나”보다 “어떻게 만들었나”를 봅니다

진단 기준은 결과물의 양이 아닙니다. 같은 결과물을 만들었더라도 요청을 어떻게 나눴는지, 받은 결과를 어떻게 확인했는지, 막혔을 때 어떻게 다시 풀었는지가 더 중요합니다.

| 축 | 측정 내용 |
|---|---|
| DECOMP (작업 분해) | 큰 일을 작은 단위로 쪼개서 요청하는지 |
| VERIFY (검증) | 받은 결과를 제대로 확인하는지 |
| ORCH (오케스트레이션) | 도구와 에이전트를 조합해서 쓰는지 |
| FAIL (실패 복구) | 막혔을 때 다시 풀어가는지 |
| CTX (컨텍스트 제공) | 배경, 제약, 예시를 충분히 주는지 |
| META (메타 인지) | 본인 사용 패턴을 스스로 인식하는지 |

각 축은 0.5 단위 7단계(L1~L7)로 진단됩니다. 점수는 절대 평가가 아니라 현재 상태를 보는 기준입니다. “L3.5보다 L4가 더 높다”는 비교는 가능하지만, “L3.5라서 부족하다”는 식으로 받아들일 필요는 없습니다. 학습 과정에서 단계가 올라가는 구조입니다.

### 워크스페이스 유형에 따라 가중치가 달라집니다

같은 사용 패턴이라도 주로 하는 일에 따라 중요도가 달라집니다. vibe-sunsang은 워크스페이스를 네 가지로 나눕니다.

| 유형 | 어떤 사람? |
|---|---|
| Builder | 코딩으로 결과물을 만드는 사람 (DECOMP·VERIFY 가중) |
| Explorer | 리서치·분석 중심 (FAIL·CTX·META 가중) |
| Designer | 기획·디자인 중심 (CTX·META 가중) |
| Operator | 자동화·운영 중심 (ORCH·FAIL 가중) |

분류는 시작 단계에서 자동으로 됩니다. 직접 고르지 않아도 됩니다.

## 진행 흐름

### 1. `/vibe-sunsang 시작` — 워크스페이스 한 번 설정

처음 한 번만 워크스페이스를 만들면 됩니다. 이후에는 다시 만들 필요가 없습니다.

```text
/vibe-sunsang 시작
```

이 명령을 입력하면 vibe-sunsang이 사용자 홈에 워크스페이스를 만들고 클로드코드 세션 기록을 스캔합니다.

```text
✓ 워크스페이스 생성: ~/vibe-sunsang/
✓ 프로젝트 스캔: ~/.claude/projects/
✓ 발견된 프로젝트에 이름 부여
✓ 워크스페이스 유형 분류 (Builder / Explorer / Designer / Operator)
✓ JSONL 로그 → Markdown 첫 변환 완료
```

권한 요청이 뜨면 `y`로 진행합니다. 작업은 본인 컴퓨터 안에서만 일어나고, 외부로 데이터를 보내지 않습니다.

### 2. `/vibe-sunsang 변환` — 대화 로그를 분석용으로 바꾸기

세션 로그는 JSONL이라 사람이 읽기 어렵습니다. 변환 명령을 내리면 마크다운으로 바꿔서 `~/vibe-sunsang/conversations/`에 저장합니다.

```text
/vibe-sunsang 변환
```

처음 변환할 때는 시간이 조금 걸릴 수 있습니다. Part 2부터 쌓인 세션이 N개 있기 때문입니다. 변환된 파일이 많아 보여도 자동으로 정리되니 그대로 두면 됩니다.

```text
✓ 클로드코드 세션 로그 스캔
✓ 변환 대상: 약 N개 세션
✓ 마크다운 변환 완료: ~/vibe-sunsang/conversations/
```

### 3. `/vibe-sunsang 멘토링` — 6축 분석 실행

이제 본 분석을 실행합니다.

```text
/vibe-sunsang 멘토링
```

`growth-analyst`라는 서브 에이전트가 변환된 로그를 읽고 6축 점수를 매깁니다. 보통 2~3분 정도 걸립니다.

```text
✓ 요청 품질 분석 (DECOMP)
✓ 검증 패턴 분석 (VERIFY)
✓ 도구·에이전트 활용 분석 (ORCH)
✓ 실패·복구 패턴 분석 (FAIL)
✓ 컨텍스트 제공 분석 (CTX)
✓ 메타 인지 분석 (META)
✓ 다음 한 걸음 추천
✓ 진단 리포트 생성: ~/vibe-sunsang/reports/mentor-2026-05-05.md
```

서브 에이전트에 맡기는 이유는 메인 컨텍스트를 보호하기 위해서입니다. 클로드코드와 진행하던 다른 작업이 있어도 이 분석과 섞이지 않습니다.

### 4. 받은 리포트에서 핵심만 정리

리포트는 길게 나올 수 있습니다. 핵심만 추려달라고 한 번 더 요청합니다.

```text
방금 받은 멘토링 리포트 핵심만 정리해줘.
- 6축 레벨
- 가장 두드러진 강점·약점 각 1개
- 발견된 안티패턴
- Part 05 시작 전 보완할 영역
```

정리 결과는 이런 형태입니다.

```text
[6축 레벨]
- DECOMP (작업 분해)  : Lv 3.5
- VERIFY (검증)        : Lv 2.5
- ORCH (오케스트레이션): Lv 3.0
- FAIL (실패 복구)     : Lv 3.5
- CTX (컨텍스트 제공)  : Lv 3.0
- META (메타 인지)     : Lv 4.0

[강점]
- 단계별로 나눠서 요청하는 패턴이 안정적 (DECOMP 강함)

[약점]
- 결과 검증이 약함 — "되는지 확인" 이후 깊이 있는 검증 부족 (VERIFY)

[발견된 안티패턴]
- "그냥 다 알아서 해줘" 류 모호한 요청 (3회 검출)
- 결과 받고 바로 다음 진행 (검증 스킵)

[Part 05 시작 전 보완할 영역]
1순위: CLAUDE.md 작성법 — CTX 약점 보완
2순위: Hook 활용법 — VERIFY 자동화
```

이 정리를 `mentor-report.md`로 저장합니다. 6축 의미가 헷갈리면 `/vibe-sunsang 지식` 명령으로 각 축의 측정 기준을 다시 확인할 수 있습니다.

### 5. Part 5 보완 영역 매핑

마지막으로, 받은 보완 영역을 Part 5의 어느 클립에서 챙길지 연결합니다.

```text
받은 보완 영역 1-2개를 Part 05의 어떤 클립에서 집중적으로 챙겨야 할지
한 줄씩 정리해줘.
```

예시는 다음과 같습니다.

```text
1순위: CLAUDE.md 작성법 (CTX 약점 보완)
→ Part 05 실습 18 (CLAUDE.md — AI에게 업무 매뉴얼 쓰기)에서 집중

2순위: Hook 활용법 (VERIFY 자동화)
→ Part 05 실습 21 (Hook — 자동화의 방아쇠 설정하기)에서 집중
```

이 내용은 `priorities-for-part05.md`로 저장합니다. `progress.json`에도 자동 기록되기 때문에 Part 5를 시작할 때 다시 불러올 수 있습니다.

## 결과물

결과물 폴더는 `50-my-work/Part04-강화하기/실습17-AI활용점검/`입니다. 여기서는 진단 리포트와 Part 5 우선순위 정리가 가장 중요합니다.

```text
실습17-AI활용점검/
├── mentor-report.md            # 6축 레벨 + 강점·약점 + 안티패턴
├── priorities-for-part05.md    # Part 5 클립별 집중 영역

~/vibe-sunsang/
├── conversations/              # 변환된 대화 로그 (보관)
└── reports/                    # 멘토링 리포트 원본
```

`완료` 또는 `/wrap`을 입력하면 `progress.json`에 다음 정보가 기록됩니다.

| 키 | 내용 |
|---|---|
| `practice_completed` | "실습 17" 추가 |
| `current_levels` | 6축 레벨 |
| `priorities_for_part05` | 보완 영역 1~2개 |
| `completed_parts` | "Part 04" 추가 |
