## 핵심 개념

### 스킬은 워크플로우를 박아둔 폴더입니다

스킬의 본체는 `SKILL.md` 한 장입니다. `CLAUDE.md`와 형식이 비슷한 마크다운 파일이에요. 다른 점은 언제 동작하는가입니다. `CLAUDE.md`는 워크스페이스 전체에서 늘 적용되는 규칙이고, 스킬은 특정 자연어 한 마디가 들어왔을 때만 깨어나는 워크플로우예요.

```text
.claude/
└── skills/
    └── deep-research/
        ├── SKILL.md         ← 본체 (워크플로우 정의)
        ├── scripts/         ← (선택) 결정론 단계용 코드
        └── references/      ← (선택) 도메인 가이드 문서
```

`SKILL.md` 맨 위에는 description 한 줄이 있습니다. 이 줄이 호출 조건을 정합니다. "딥리서치", "깊이 조사", "여러 출처로 정리" 같은 자연어가 들어오면 클로드코드가 description을 보고 스킬을 가져옵니다. 사용자가 슬래시를 칠 필요도, 폴더 이름을 외울 필요도 없어요.

작동 흐름은 짧습니다.

```text
사용자 자연어 한 마디
    ↓
클로드코드가 등록된 스킬들의 description 스캔
    ↓
가장 잘 맞는 스킬 1개 선택 → SKILL.md 본문 로드
    ↓
SKILL.md에 적힌 단계대로 작업 진행
```

설치는 워크스페이스 안에 폴더 하나 만들면 끝납니다. `~/fastcampus-cc/.claude/skills/스킬이름/SKILL.md` 형태예요. 이 폴더만 다른 컴퓨터로 옮기면 같은 스킬이 그대로 동작합니다.

### 왜 "클로드코드의 꽃"이라 부르는가 — 3 가치

스킬을 클로드코드의 꽃이라 부르는 이유는 한 가지예요. 본인 업무에 가져갔을 때 효과가 가장 큰 단위라서. 그 효과를 세 갈래로 풀어볼게요.

**생산성** — 30분 걸리던 작업이 5분으로 줄어듭니다.

같은 프롬프트를 매번 새로 짜본 경험 있으신가요? 매일 똑같은 반복 업무를 이제는 스킬로 만들어 자동화하세요. Part 06 마지막 챕터에서 만들 콘텐츠 파이프라인을 예로 들면, 리서치·이미지·HTML·음성·영상까지 6단계를 한 번에 만듭니다. 따로따로 했을 때 30분 걸리던 일이 이젠 5분이면 끝낼 수 있어요.

**재현성** — 같은 입력에 같은 결과가 나옵니다.

AI에게 같은 일을 시켜도 매번 조금씩 결과가 다릅니다. 단어 선택이 달라지고, 어떤 항목은 빠지기도 해요. 스킬은 단계마다 AI에게 맡길지 스크립트로 빼둘지를 나눌 수 있어서, 자유도와 안정성을 같이 잡습니다. API 호출이나 JSON 파싱처럼 답이 정해진 단계는 스크립트로 빼고, 해석이나 요약처럼 자유도가 필요한 단계는 SKILL.md 본문에 둡니다.

**자산화** — 노하우가 팀에 남습니다.

머리로만 생각하고 끝나는 워크플로우는 금방 사라집니다. 생각한 걸 자산으로 만들어야 의미가 있어요. 그래서 스킬이 가치가 있다고 생각합니다. 스킬은 본인 작업 순서를 마크다운 한 장에 박아두는 형태라, 팀원이 같은 폴더를 받아 쓰면 본인 노하우가 그대로 재현됩니다. 시간이 지나면 한 사람이 만든 스킬이 팀 전체 표준 작업이 되기도 해요.

셋 중 무엇이 먼저 와닿는지는 사람마다 다릅니다. 마케팅 일을 하는 분은 생산성, 운영 담당자는 재현성, 컨설팅 일을 하는 분은 자산화를 먼저 본다는 식이에요.

### 메타 흐름 — "그냥 만들어줘"는 안 통합니다

스킬을 잘 만드는 사람과 못 만드는 사람 차이는 첫 프롬프트 한 줄에서 갈립니다.

```text
나쁜 시작: "딥리서치 스킬 만들어줘"
좋은 시작: "어떤 주제에 대해 깊이 조사해서 정리하려는데,
          어떻게 워크플로우를 구성해야 할까?"
```

차이가 작아 보이지만 결과는 크게 갈려요. 앞 문장은 클로드코드가 곧장 SKILL.md를 쓰기 시작합니다. 본인이 그동안 알고 있던 "딥리서치"의 한계가 그대로 반영됩니다. 뒤 문장은 클로드코드가 먼저 "딥리서치를 어떻게 하는지" 그 자체를 조사합니다. 다관점 출처 수집, 교차 검증, 종합, 출처 명시 같은 단계가 펼쳐져요. 그중에 본인이 모르던 단계가 들어옵니다.

그래서 Part 6의 모든 실습은 4단계 메타 흐름을 따릅니다.

```text
[1단계] 워크플로우 잡기 (방법론 조사)
   "[하려는 일] 만들려는데, 어떻게 워크플로우를 구성해야 할까?"

[2단계] 보완 + 실행 가능 단계 정의
   정리된 단계들 같이 살펴보고 보완
   → AI에게 맡길 단계와 코드로 분리할 단계를 여기서 구분

[3단계] 스킬로 만들기
   정의된 흐름을 SKILL.md로 정리

[4단계] 만든 스킬로 테스트·검증
   실제로 호출해서 description 매칭 + 결과 확인 + 부족한 부분 손보기
```

이 흐름이 9개 실습 모두에 들어 있습니다. 같은 흐름을 9번 반복하는 이유는 단순해요. Part 6을 마친 뒤 본인 일에 새 스킬을 만들 때, 이 4단계가 손에 익어야 하기 때문입니다. SKILL.md 작성 자체는 클로드코드가 합니다. 본인이 가져가야 할 건 "어떻게 시작 프롬프트를 던질까"입니다.

## 결과물 저장 규칙

Part 5와 같은 규칙입니다. 두 곳에 나눠 저장됩니다.

```text
강의 워크스페이스 (활성 스킬)
└── ~/fastcampus-cc/.claude/skills/{스킬이름}/SKILL.md
    → 호출 즉시 동작하는 본체

학습 결과물 (진도 + 원본 사본)
└── ~/fastcampus-cc/50-my-work/Part06-스킬만들기/실습{NN}-{제목}/
    ├── README.md                  ← 진도 메타 (자동 작성)
    └── skill-source/SKILL.md      ← 만든 스킬의 원본 사본
```

활성 스킬은 `.claude/skills/` 하위에 있어야 클로드코드가 발견합니다. `50-my-work/` 안의 사본은 진도 기록용이에요. 두 곳에 같은 파일이 있다고 헷갈리지 않으셔도 됩니다. 한쪽은 동작용, 한쪽은 기록용입니다.

각 클립을 끝낼 때 `완료` 또는 `/wrap`을 입력하면 스킬이 알아서 정리합니다. `README.md` 생성, `progress.json` 업데이트, 다음 클립 안내까지 한 번에 챙겨줘요.
