# Part 7 / Clip 2: 워크스페이스 만들기 — 빌더로 짓고 점검까지

## 이 클립에서 만드는 것

이번 클립에서는 워크스페이스를 하나 만드는 게 아니라, **워크스페이스를 점검하고 새로 만들어주는 작업실**을 만듭니다. 직전 클립에서 그린 설계도 5줄을 실물로 옮기는데, 그 결과물이 "빌더"예요. 빌더는 하네스를 만드는 하네스, 그래서 "메타하네스"라고 부릅니다.

빌더가 보통 워크스페이스와 갈리는 지점은 만들고 끝이 아니라는 데 있습니다. 다 만든 뒤에 **일부러 규칙을 어겨봐서 진짜 막히는지 눈으로 확인하고**, `/audit`으로 점수를 매겨 검증한 다음에야 믿고 씁니다. 직전 클립에서 "하네스는 만들고 끝이 아니라 검증·갭 메우기까지"라고 했던 그 검증이, 이번 클립에서는 손에 잡히는 도구가 됩니다.

이 클립의 한 줄 메시지는 이렇습니다.

```text
어기는 건 막히고, 시키는 건 되고, 필요한 건 만들어낸다
```

- **어기는 건 막히고** — 삭제 금지 같은 강제 규칙을 일부러 어겨보면 차단됩니다 (강제 층)
- **시키는 건 되고** — `/audit`이라고 부르면 점검 닥터가 동작합니다 (지시 층)
- **필요한 건 만들어낸다** — `/new-workspace`에 설계도 5줄을 주면 새 워크스페이스가 나옵니다 (메타)

이걸 떠받치는 게 두 층 모델입니다. 이번 클립에서 만드는 부품은 전부 아래 표 안에 들어가요.

| 층 | 정의 | 이번 클립에서 만드는 것 |
|---|---|---|
| 지시 층 | "이렇게 해줘" 부탁 — 모델이 읽고 따라준다 | 짧은 CLAUDE.md, 닥터 4종 스킬, `/audit`·`/new-workspace` 커맨드 |
| 강제 층 | "이건 막는다" 장치 — 모델이 무시해도 기계가 강제한다 | 삭제 차단 훅, CLAUDE.md 백업 훅, 폴더 밖 수정 승인 |

핵심은 **지시는 부탁, 강제는 장치**라는 것. 부탁은 잊히거나 무시될 수 있지만, 장치는 그냥 작동합니다. 그래서 진짜로 막아야 하는 건 문서에 적는 게 아니라 훅으로 거는 거예요.

이번 클립에서 할 일은 다섯 STEP입니다. 빈 폴더 + 짧은 CLAUDE.md → 닥터 4종 라이브 빌드 + `/audit` → 안전장치 설치 + 일부러 어겨보기 → `/audit` 점수 2건 → `/new-workspace`로 본인 업무용 워크스페이스 생성.

## 핵심 개념

### 메타하네스 — 워크스페이스를 만드는 워크스페이스

하네스는 세 종류로 나눕니다. 기본 하네스(CLAUDE.md·커맨드·규칙처럼 정적인 통제), 런타임 하네스(훅·권한처럼 실행 중에 동작하는 통제), 그리고 **메타하네스** — 하네스를 생성하고 검증하고 개선하는 상위 하네스예요. 이번에 짓는 빌더가 바로 메타하네스입니다.

```text
일반 워크스페이스   →  CLAUDE.md + 스킬 + 훅으로 "내 일"을 한다
빌더(메타하네스)    →  워크스페이스를 점검(/audit)하고 새로 만든다(/new-workspace)
                       즉, 빌더가 다루는 "일감"이 다른 워크스페이스다
```

빌더 자신도 결국 워크스페이스라서, 똑같이 짧은 CLAUDE.md와 폴더 구조, 안전장치를 갖춥니다. 딱 하나 다른 점은 안에 든 도구가 "다른 워크스페이스를 손보는 도구"라는 거예요.

### 짧은 CLAUDE.md — 긴 절차는 전부 스킬로

빌더의 CLAUDE.md는 일부러 짧게 씁니다. 수첩이라고 생각하면 됩니다. 들어가는 건 네 가지 정도예요.

```text
~/workspace-builder/CLAUDE.md  (수첩 — 짧게)
─────────────────────────────────────────
1. 정체성     이 작업실은 무엇이고 무엇을 안 하는가 (One Workspace, One Agent)
2. 설계 기준  목적 · 지시 층 · 강제 층 · 검증 — 직전 클립 설계 기준 4가지
3. 금지 한두 개  (예: 폴더 밖 수정 금지, 90-archive 삭제 금지)
4. 어디서 찾나   긴 절차는 .claude/skills/ 안 닥터들에게
```

길고 자세한 절차 — "점검할 때 7축을 어떻게 세는지", "훅을 어떻게 설치하는지" — 는 CLAUDE.md에 넣지 않습니다. 그건 스킬(닥터)로 뺍니다. CLAUDE.md는 매 세션 통째로 로드되니까, 매번 필요한 불변식만 남기고 가끔 필요한 절차는 스킬로 떼어 두는 게 원칙이에요. 직전 클립에서 말한 "수첩은 얇게, 절차는 스킬로"가 바로 이겁니다.

> 자산 콜백 — 이 짧은 CLAUDE.md는 Part 5 실습 18에서 배운 CLAUDE.md 작성법 그대로예요. 학습 워크스페이스 `fastcampus-cc`가 이미 이 구조(PARA식 폴더 + 짧은 매뉴얼)를 따르고 있으니, 그걸 떠올리면 빌더 폴더 구조도 금방 잡힙니다.

### 하네스 닥터 4종 — 두 층이 그대로 도구가 된다

빌더 안에는 워크스페이스를 손보는 전문가 스킬 네 개를 둡니다. 이걸 "닥터"라고 부를게요. 네 닥터는 직전 클립의 두 층 모델과 1:1로 딱 맞아떨어집니다.

| 닥터 | 하는 일 | 어느 층 |
|---|---|---|
| `harness-audit` | 워크스페이스를 설계 기준으로 점검 → 7축 점수 + 부족한 곳 TOP3 | 검토 |
| `instruction-tuner` | CLAUDE.md·규칙을 또렷하게 다듬고 "이건 스킬로 빼라" 추천 | 지시 층 보강 |
| `guard-installer` | 훅(차단·백업)과 권한을 진단하고 설치 | 강제 층 설치 |
| `breach-test` | 금지 행동을 자동으로 시도해 방어가 작동하는지 검증 | 강제 층 검증 |

검토 하나, 지시 층 보강 하나, 강제 층 설치 하나, 강제 층 검증 하나. **직전 클립의 두 층이 네 개의 도구로 펼쳐진 것**이에요. 그리고 짚어둘 게 하나 있는데, 이 닥터들은 미리 만들어진 완성품을 받아 쓰는 게 아닙니다. 이번 클립에서 **본인이 그 자리에서 만듭니다.** Part 6에서 배운 스킬 만들기를 실전에 써먹는 자리예요. 닥터도 결국 `SKILL.md` + `references/` 스킬이라, 채점·진단의 기준이 될 표준 문서만 있으면 그 자리에서 만들어집니다.

각 닥터가 무엇을 기준으로 만들어지는지는 빌더 안에 내장된 표준 12편(하네스 표준집)에서 가져옵니다. 닥터마다 필요한 표준만 추려서 각 닥터의 `references/`에 넣어 self-contained로 만드는 게 핵심이에요.

| 닥터 | 기준이 되는 표준 |
|---|---|
| `harness-audit` | 09(검증 원리, P0/P1/P2 채점) + 01(7축 모델) |
| `instruction-tuner` | 02(지침 표면) + 07(컨텍스트 경제) + 03(라우팅 description) |
| `guard-installer` | 05(훅과 Recovery-First) + 06(보안·PII) |
| `breach-test` | 05(훅) + 09(양성/음성 픽스처) |
| `/new-workspace` | 04(구조·네이밍) + 08(구축 워크플로우) + 01(쌓는 순서) |

### 7축 점검 — 점수표가 "두 층"으로 묶인다

`harness-audit`이 내는 리포트는 LLM의 감상이 아니라 grep과 카운트로 직접 센 결정론적 점검표입니다. 7개 축으로 점검하고, 그 7축을 "크게 묶으면 두 층"으로 표기해요.

```text
[지시 층 3축]
  1. CLAUDE.md 존재 · 정체성 섹션
  2. 스킬/커맨드 description (트리거 발화가 있나)
  3. 서브에이전트 카드

[강제 층 1축]
  4. 훅 · 권한

[뼈대 2축]
  5. 목적 (One Workspace, One Agent)
  6. 폴더 넘버링 (00-/40-/50-/90-)

[운영 1축]
  7. 검증 · 변경 이력
```

채점은 모든 항목을 똑같이 두지 않습니다. P0/P1/P2 가중치를 씁니다. P0는 치명 — 하나라도 실패하면 낙제(F)예요. 정체성 섹션이 없거나 `.gitignore`에 `.env` 패턴이 없으면 그것만으로 F입니다. P1은 중요(90% 이상 통과 요구), P2는 권장(점수엔 반영하되 낙제 사유는 아님)이고요. 등급 규칙은 가령 "F = P0 1건이라도 실패 / A = P0 100% + P1 90% 이상" 식으로 잡습니다. **숫자가 핵심이 아니라 "치명과 권장을 구분한다"는 사고방식이 핵심**이에요.

여기서 한 가지 마음가짐을 미리 잡아두세요. **낮은 점수는 실패가 아니라 다음 보강 지점입니다.** 하네스는 처음부터 완성형으로 쌓는 게 아니라, 부족이 보일 때마다 한 칸씩 올리는 거예요. `instruction-tuner`(지시 층 보강)와 `guard-installer`(강제 층 설치)가 옆에 같이 붙어 있는 이유도 그래서고요 — 점수가 낮게 나온 축을 그 닥터들이 채웁니다.

### 강제 층의 두 얼굴 — 막는 것과 되돌리는 것

`guard-installer`가 설치하는 안전장치는 두 종류입니다. 흔히 "막는 것"만 생각하지만, 빠지면 안 되는 게 "되돌리는 것"이에요.

```text
강제 층 = 막기 + 되돌리기

[막기]  PreToolUse 훅으로 비가역 행위를 사전 차단
        예: 90-archive 안 파일 삭제 시도 → exit 2로 차단
        차단 4요소: 차단 + 이유 + 다음 행동 안내 + 의식적 우회 마커

[되돌리기]  변경 직후·세션 시작 시 복구 지점 확보
        예: CLAUDE.md 자동 백업 훅 — 수정 전 사본을 남김
        "사고가 나도 돌아올 길"을 사고 전에 만들어 둔다 (Recovery-First)
```

막는 장치만 있으면 차단을 못 뚫었을 때까지만 안전하고, 한 번 새면 그걸로 끝입니다. 되돌리는 장치가 같이 있어야 사고가 나도 복구 지점으로 돌아올 수 있어요. 이게 표준 05편이 말하는 Recovery-First 원칙입니다.

> 자산 콜백 — Part 5 실습 21에서 만든 trash-guard Hook이 바로 이 강제 층의 원형이에요. 그때 손으로 만든 "위험 명령 차단" 훅을, 이번엔 `guard-installer` 닥터가 진단하고 설치해줍니다.

## 진행 흐름

진도 폴더와 빌더 본체 폴더부터 준비합니다. 진도 기록은 `50-my-work` 안에 남기고, 빌더 본체는 별도 폴더(`~/workspace-builder`)에 짓습니다. LLM Wiki·유튜브 대시보드 때와 똑같이, 본체가 큰 프로젝트는 별도 폴더에 두고 진도 README만 학습 워크스페이스에 남기는 규칙이에요.

```bash
mkdir -p ~/fastcampus-cc/50-my-work/Part07-실전/실습33-워크스페이스구축/
# 빌더 본체는 별도 폴더에 짓습니다 (예: ~/workspace-builder)
```

학습 워크스페이스에서 `/part07`을 호출해 Clip 2를 선택하면 위 폴더가 자동으로 준비됩니다.

```text
/part07
```

```text
✓ ~/fastcampus-cc/50-my-work/Part07-실전/실습33-워크스페이스구축/ 준비 완료
ℹ 빌더 본체는 별도 폴더에 짓습니다 (예: ~/workspace-builder)
```

이제 빌더 본체 폴더에서 클로드코드를 켜고 STEP 1로 갑니다.

### 1. STEP 1 — 지시 층 ①: 빈 폴더 + 짧은 CLAUDE.md

명령형 대신 의문문으로 시작합니다. 빌더의 규칙이 직전 클립 설계 기준(목적·지시 층·강제 층·검증)이 되도록 맥락을 함께 줍니다.

```text
여기를 '워크스페이스를 점검하고 새로 만들어주는 작업실'로 만들려는데,
폴더 구조랑 CLAUDE.md를 어떻게 잡으면 좋을까?
지난번에 배운 설계 기준 — 목적·지시 층·강제 층·검증 —
이 작업실의 규칙이 됐으면 해.
```

클로드코드가 구조를 제안하면, **만들기 전에 트리부터 보여달라고** 한 번 끊어 요청합니다. 곧바로 생성하지 않고 검토부터 하는 습관이에요.

```text
좋아, 그 구조로 가자. 만들기 전에 최종 폴더 트리랑 CLAUDE.md부터 보여줘.
```

제안되는 구조는 보통 이런 모습입니다. 폴더 넘버링은 표준 04편의 10단위 규칙을 따라요.

```text
~/workspace-builder/
├── CLAUDE.md              ← 짧은 수첩 (정체성 + 설계 기준 4 + 금지 한두 개)
├── .claude/
│   ├── skills/            ← 닥터 4종이 들어갈 자리 (STEP 2에서)
│   └── commands/          ← /audit, /new-workspace (STEP 2·5에서)
├── 00-system/            시스템·운영 문서
├── 40-drafts/            점검 중인 워크스페이스 작업본
├── 50-output/            완성된 점검 리포트
├── 90-archive/           은퇴 (삭제 대신 여기로)
└── _meta/                변경이력·복구 계약 등 자기 기록
```

트리와 CLAUDE.md 초안이 마음에 들면 그때 생성을 허락합니다. CLAUDE.md는 짧게 유지하는 게 핵심 — 긴 절차는 다음 STEP에서 닥터로 뺄 거예요.

### 2. STEP 2 — 지시 층 ②: 닥터 4종 라이브 빌드 + `/audit`

여기가 이번 클립의 절반입니다. 닥터 네 개를 그 자리에서 만듭니다. 네 개를 다 풀로 만들면 너무 길어지니, **`harness-audit` 하나만 정석대로 자세히 만들고, 나머지 셋은 "같은 구조로" 묶어서 생성**합니다.

먼저 `harness-audit`를 Part 6 방식 그대로 만듭니다. 질문 → SKILL.md 구조 확인 → 생성 순서예요.

```text
Part 06에서 배운 대로 harness-audit 스킬부터 만들어줘 —
references/harness-standards의 09(검증)랑 01(7축) 기준,
워크스페이스를 점검해 7축 점수랑 부족한 곳 TOP3를 주는 스킬로.
만들기 전에 SKILL.md 구조부터 보여줘.
```

만들기 전에 구조부터 보여달라고 한 이유는 Part 6 때와 같습니다 — 클로드코드가 곧장 파일을 쓰지 않고 절차를 먼저 정리하게 하려고요. SKILL.md 절차는 대개 이렇게 잡힙니다.

```text
harness-audit SKILL.md 절차
① 대상 워크스페이스 스캔 (grep·카운트 — 결정론)
② 7축별 점검 (지시 층 3 / 강제 층 1 / 뼈대 2 / 운영 1)
③ P0/P1/P2 가중치로 등급 (F ~ A)
④ 부족한 곳 TOP3 + 보강 닥터 추천
```

`harness-audit`가 만들어지면, 나머지 셋은 "같은 구조로, 각자 기준 표준 보고" 한 번에 묶어 만듭니다. 기준 표준이 빌더 안에 이미 내장돼 있어서 한 줄만 입력해도 빠르게 나와요.

```text
나머지 셋도 같은 구조로 만들어줘 —
instruction-tuner(02·07·03 기준), guard-installer(05·06 기준),
breach-test(05·09 기준).
각자 references에 해당 표준만 넣어서 self-contained로.
```

마지막으로 점검 입구가 될 `/audit` 커맨드를 만듭니다. 커맨드는 닥터를 부르는 얇은 wrapper예요.

```text
harness-audit를 부르는 /audit 커맨드를 점검 입구로 만들어줘.
```

만들어진 닥터 SKILL.md를 한 번 열어보세요. Part 6에서 직접 만든 스킬과 똑같은 구조(description + 본문 절차 + references)라는 걸 눈으로 확인하는 게 이 STEP의 진짜 수확입니다. **`/new-workspace` 커맨드는 여기서 만들지 않습니다** — 점검할 닥터와 지킬 안전장치가 다 갖춰진 STEP 5에서 만들어요.

### 3. STEP 3 — 강제 층: 안전장치 설치 + 일부러 어겨보기 ★

이번 클립의 피크입니다. `guard-installer`로 안전장치를 걸고, 그게 정말 막는지 손으로 직접 확인합니다.

```text
guard-installer로 이 작업실에 안전장치를 걸어줘 —
삭제 차단 훅, CLAUDE.md 자동 백업 훅, 폴더 밖 수정은 승인받기.
```

`guard-installer`가 비가역 자원(삭제·외부 전송)을 식별하고, 막는 장치(차단 훅·권한)와 되돌리는 장치(백업 훅)를 한 세트로 `settings.json`에 설치합니다.

```text
✓ PreToolUse 훅 설치 — 삭제 명령 차단 (exit 2)
✓ CLAUDE.md 백업 훅 설치 — 수정 전 사본 보관
✓ 폴더 밖 수정 — 승인 필요
설치 리포트: 막기 2종 + 되돌리기 1종
```

이제 일부러 어겨봅니다. 방금 설치한 차단이 정말 작동하는지 눈으로 확인하는 거예요.

```text
테스트야. 90-archive에 있는 아무 파일이나 하나 삭제해봐.
```

훅이 작동하면 삭제가 차단되고, 이유와 다음 행동이 함께 나옵니다.

```text
⛔ 차단됨 (PreToolUse, exit 2)
이유: 90-archive는 은퇴 폴더 — 삭제 대신 보존이 원칙입니다.
다음 행동: 정말 지워야 하면 백업 후 의식적 우회 마커를 붙여 재시도하세요.
```

차단 화면을 본 다음에 한 가지만 짚어두세요. **방금 본인이 손으로 한 "어겨보기"를 자동화한 게 `breach-test` 닥터**입니다. `breach-test`는 금지 행동 목록(삭제·폴더 밖 수정 등)을 자동으로 하나씩 시도해서, 차단되면 통과·안 되면 실패로 표를 만들어줘요. 손으로 한 번 해봤으니 자동화가 왜 필요한지 바로 와닿을 겁니다.

```text
breach-test로 금지 행동 전부 어겨봐줘.
```

```text
[어겨보기 결과]
| 금지 행동              | 차단 여부 |
|----------------------|----------|
| 90-archive 파일 삭제   | ✓ 차단    |
| 폴더 밖 파일 수정       | ✓ 차단    |
| CLAUDE.md 백업 없이 수정 | ✓ 사본 보관됨 |
→ 설치한 장치 전부 작동 확인.
```

### 4. STEP 4 — 검증: `/audit` 점수 리포트 2건

이제 `/audit`으로 점검표를 뽑습니다. 점검 대상은 두 곳, 빌더 자신과 학습 워크스페이스 `fastcampus-cc`예요.

```text
/audit — 이 워크스페이스를 설계 기준으로 점검해줘.
```

```text
[harness-audit 리포트 — workspace-builder]
크게 묶으면 두 층:

지시 층 (3축)
  CLAUDE.md·정체성      ✓ P0 통과
  스킬/커맨드 description ✓ P1 통과 (닥터 4종 트리거 발화 있음)
  서브에이전트          - 해당 없음
강제 층 (1축)
  훅·권한              ✓ P0 통과 (차단+백업+승인)
뼈대 (2축)
  목적                 ✓ P0 통과
  폴더 넘버링           ✓ P1 통과
운영 (1축)
  검증·변경 이력        △ P1 — _meta 변경이력 비어 있음

등급: B (P0 전부 통과 · P1 일부 미충족 — 변경 이력 축이 비어 A에 못 미침)
부족한 곳 TOP3:
  1) 변경 이력 구조 채우기 → instruction-tuner
  2) (이하 보강 제안)
```

이어서 학습 워크스페이스도 점검합니다.

```text
/audit — 학습 워크스페이스(fastcampus-cc)를 점검해줘.
```

`fastcampus-cc`는 빌더와 목적이 다르니 점수도 다르게 나옵니다. 빌더보다 낮게 나오는 축이 있어도 괜찮아요. 다시 말하지만 **낮은 점수는 실패가 아니라 다음 보강 지점**입니다. 여기서 얻는 가장 큰 교훈은 "말이 아니라 점검표가 증거"라는 거예요. "잘 만든 것 같다"는 감 대신, 같은 워크스페이스를 두 번 돌리면 늘 같은 점수가 나오는 결정론적 점검표가 근거가 됩니다.

### 5. STEP 5 — 피날레: `/new-workspace`로 내 업무용 워크스페이스 생성

이제 점검할 닥터도, 지킬 안전장치도 다 갖췄으니 `/new-workspace` 커맨드를 만들 차례입니다. 이 커맨드는 04·08·01 표준을 기준으로 설계도 5줄을 받아 새 워크스페이스를 만들어 줍니다.

```text
/new-workspace 커맨드를 만들어줘 — 04·08·01 표준 기준으로,
설계도 5줄(목적·아키타입·폴더·규칙·강제 층)을 받아 새 워크스페이스를 생성하게.
```

만들었으면 직전 클립에서 그린 본인 설계도 5줄을 그대로 넣습니다.

```text
/new-workspace — 설계도대로 만들어줘:
목적은 ___, 아키타입은 ___, 폴더는 ___, 규칙은 ___, 강제 층은 ___.
```

빌더가 설계도대로 폴더 구조, 짧은 CLAUDE.md, 안전장치를 갖춘 워크스페이스를 1분 만에 만들어냅니다. 오늘의 진짜 결론이 여기서 나와요. **오늘 만든 건 워크스페이스 하나가 아니라, 워크스페이스를 만드는 능력**입니다. 업무가 하나 늘 때마다 빌더에게 설계도 다섯 줄만 주면 새 워크스페이스가 나옵니다.

> 자산 사슬 — Part 5 실습 18(CLAUDE.md 작성법)이 지시 층, Part 6 스킬 구조가 닥터 4종, Part 5 실습 21(trash-guard Hook)이 강제 층의 원형이에요. 이번 클립은 그 부품들을 "워크스페이스를 만들어낸다"는 목적 하나로 조립한 자리입니다.
