# 00. 시작하기 전에

> Claude Code를 시작하기 전에 가장 많이 받는 질문 4가지에 답합니다.

---

## Q1. 클로드 코드랑 코워크(Cowork)랑 뭐가 달라요?

한 줄로 말하면, <strong>Claude Code는 만드는 도구</strong>이고, <strong>Cowork는 처리하는 도구</strong>입니다.

| | Claude Code | Claude Cowork |
|---|---|---|
| **하는 일** | 새로운 걸 만듦 (앱, 문서, 자동화) | 기존 업무를 처리함 (메일, 문서 정리) |
| **비유** | AI 건축가 | AI 비서 |
| **작동 방식** | 터미널에서 텍스트 명령 | 샌드박스(격리 환경) 안에서 처리 |
| **대상** | 터미널 사용에 열려 있는 사람 | 터미널을 전혀 모르는 사람 |

### Cowork의 샌드박스 — 안전하지만 느리고 비싼 이유

Cowork는 내 컴퓨터 안에 만들어진 <strong>격리된 가상 환경(로컬 VM)</strong>에서 실행됩니다. 이 샌드박스 덕분에 안전하지만, 구조적인 한계가 있습니다.

**토큰 소비가 큽니다.** Cowork가 화면을 조작할 때는 스크린샷을 찍어서 AI가 분석하는 방식을 씁니다. 텍스트 한 줄이면 될 걸 이미지로 처리하니 토큰 소비량이 훨씬 큽니다. Claude Code는 텍스트로 직접 파일을 읽고 쓰기 때문에 같은 작업을 훨씬 적은 토큰으로 처리합니다.

**동작 속도가 느립니다.** 화면 조작 시 스크린샷 촬영 → AI 분석 → 클릭 → 화면 대기를 매번 반복합니다. Claude Code는 명령어 실행 → 텍스트 결과 반환이라 훨씬 빠릅니다.

**접근이 제한됩니다.** 격리된 환경이라 로컬 파일에 직접 접근할 수 없고, 설치할 수 있는 소프트웨어도 제한됩니다. Claude Code는 내 컴퓨터 전체에 접근할 수 있어 자유도가 높습니다.

> **Cowork가 존재하는 이유**: 능력의 차이가 아니라 "터미널을 열 수 있느냐"의 차이입니다. 그 허들을 넘은 분에게는 Claude Code가 압도적입니다.

## Q2. 데스크톱 앱이나 VS Code 플러그인이 있는데 터미널을 꼭 써야 하나요?

데스크톱 앱과 VS Code 플러그인이 처음 나왔을 때는 터미널에 비해 기능이 제한되고 신규 기능 적용도 늦었습니다. 하지만 2026년 4월 기준으로 <strong>대부분의 기능이 모두 적용</strong>되었고, 같은 엔진을 쓰기 때문에 MCP, Hooks, Skills, 플러그인 등이 어디서든 거의 동일하게 동작합니다.

**그럼에도 터미널을 권장하는 이유가 있습니다.**

### 멀티세션 — 생산성의 핵심

Claude Code로 생산성을 제대로 높이려면 <strong>여러 세션을 동시에 돌리는 것</strong>이 필수적입니다.

예를 들어 터미널 창을 4개 띄워서:
- 창 1: 프론트엔드 작업
- 창 2: 백엔드 API 개발
- 창 3: 테스트 실행
- 창 4: 문서 작성

이런 식으로 4개의 Claude Code가 <strong>동시에 각자의 일을 처리</strong>합니다. 하나가 끝나면 바로 다음 지시를 내리면 됩니다.

데스크톱 앱이나 VS Code 플러그인에서도 세션을 여러 개 만들 수 있지만, <strong>한 화면에 동시에 펼쳐놓고 모니터링하는 건 터미널에서만 가능</strong>합니다. tmux 같은 도구를 쓰면 한 화면을 패널로 나눠서 모든 세션의 진행 상황을 한눈에 볼 수 있습니다.

### 터미널만 할 수 있는 것들

**자동화와 확장성.** 터미널이 단순히 "더 빠르다"가 아니라, 터미널에서**만** 가능한 영역이 있습니다:

- **파이프 스크립팅**: `cat error.log | claude -p "원인 분석"` — 다른 도구의 출력을 Claude에 바로 먹이기
- **CI/CD 통합**: GitHub Actions에서 PR 올라올 때마다 자동으로 코드 리뷰
- **정기 자동화**: cron으로 매일 아침 의존성 검사를 돌리기
- **원격 실행**: SSH로 서버에 접속해서 `claude -p "로그 분석해줘"` 실행
- **백그라운드 실행**: 터미널을 닫아도 세션이 계속 돌아감
- **커뮤니티 도구**: VibeProxy(다른 AI 모델 연동), claude-esp(내부 동작 모니터링) 등 터미널 기반 오픈소스 생태계

이런 것들은 데스크톱 앱이나 VS Code의 GUI 환경에서는 구조적으로 불가능합니다.

### 안정성

2026년 4월 기준, 기능은 거의 동일하지만 <strong>안정성에서는 아직 차이</strong>가 있습니다:

| | 터미널 (CLI) | 데스크톱 앱 | VS Code 플러그인 |
|---|---|---|---|
| **Hooks** | 완전 동작 | PostToolUse 버그 보고 | 버그 보고 있음 |
| **MCP 서버** | 안정적 | Connectors UI 지원, 다수 연결 시 성능 저하 | 연결 버그 보고 있음 |
| **쾌적성** | 가볍고 빠른 반응 | 상대적으로 무거움 | 에디터 위에서 실행되어 무거움 |

이 차이는 빠르게 좁혀지고 있지만, 현재 시점에서 가장 안정적인 환경은 터미널입니다.

### 터미널이라는 허들

솔직히 말하면, 많은 분들이 <strong>"터미널은 어려워 보이니까 데스크톱 앱으로 충분하겠지"</strong>라고 생각하며 멈춰 있습니다. 익숙한 GUI 환경에서 벗어나기 싫은 마음을 "이 정도면 충분해"라고 합리화하는 거죠.

하지만 AI 시대에 터미널은 더 이상 개발자만의 도구가 아닙니다. <strong>자연어로 말하면 되는 시대</strong>가 왔고, Claude Code가 바로 그 증거입니다. 터미널을 여는 것 — 그 한 번의 허들만 넘으면 할 수 있는 것의 범위가 완전히 달라집니다.

> 그 한계를 꼭 깨보시길 바랍니다.

## Q3. 클로드 코드 설치했는데 뭐부터 하면 되나요?

"해달라"고 하기 전에, <strong>내가 뭘 하고 싶은지</strong>부터 찾으세요.

### 먼저, 활용 사례를 찾아보세요

Claude Code로 뭘 할 수 있는지 감이 안 잡히는 건 당연합니다. 처음엔 다들 그렇습니다. 그래서 바로 뭔가를 만들려고 하기보다, <strong>다른 사람들이 어떻게 쓰고 있는지</strong>를 먼저 둘러보세요.

- SNS, 유튜브, 블로그에서 "Claude Code 활용" 사례를 검색해보기
- 바이브코딩 커뮤니티에서 사람들이 만든 것들 구경하기
- "이거 나도 해보고 싶다"는 게 생기면 그때 시작하기

### 그다음, Claude와 대화하면서 정의하세요

하고 싶은 게 어렴풋이 잡히면, 바로 "만들어줘"라고 하지 마세요. 먼저 **Claude에게 물어보세요**:

```
"나는 이런 걸 하고 싶은데, 너는 어떻게 할 수 있어?"
"이런 프로젝트를 만들려면 뭐가 필요해?"
"이 분야에 대해 리서치 좀 해줘"
```

Claude Code는 단순히 시킨 걸 수행하는 도구가 아닙니다. <strong>같이 생각하고, 조사하고, 방향을 잡아주는 파트너</strong>입니다. 내가 뭘 원하는지 대화를 통해 구체화하고, 그 과정에서 Claude가 가능한 것과 불가능한 것을 알려줍니다.

> 설치 방법, 기본 명령어, CLAUDE.md 설정 같은 것들은 본 가이드의 <strong>섹션 1</strong>부터 순서대로 다루고 있습니다. 뭘 하고 싶은지가 정해지면 그때 읽어도 늦지 않습니다.

## Q4. Pro 요금제($20)면 충분한가요?

솔직하게 말씀드리면, <strong>다른 AI 도구에 비해 Claude Code는 비용이 높은 편</strong>입니다. 그리고 Pro 요금제만으로는 Claude Code를 제대로 활용하기 어렵습니다.

### Pro($20)의 현실

Pro의 5시간 사용량 한도는 생각보다 금방 찹니다. 집중해서 작업하면 **1시간 만에 한도에 도달하는 경우도 빈번합니다.** AI로 원하는 결과를 얻기도 전에 토큰이 다 차서 멈추고, 한도가 풀릴 때까지 기다리는 상황이 반복됩니다.

- 가장 강력한 <strong>Opus 모델은 사실상 사용이 불가능</strong>합니다. 한도가 금방 차버립니다.
- 기본 모델인 Sonnet으로도 간단한 작업은 가능하지만, 복잡한 프로젝트에서는 한계가 있습니다.

### Max($100)를 꼭 한번 경험해보세요

Claude Code의 진짜 생산성은 **고급 기능을 조합할 때** 나옵니다:

- **멀티세션**: 여러 Claude를 동시에 돌려 병렬 작업 → 토큰을 빠르게 소모
- **에이전트 팀**: 여러 AI가 역할을 나눠 협업 → 토큰을 수배 이상 소모
- **Opus 모델**: 복잡한 아키텍처 설계, 디버깅에 필수 → Pro에서는 사실상 불가

이런 기능들을 활용하려면 Pro의 5배 사용량이 제공되는 <strong>Max $100 이상이 필수적</strong>입니다.

Pro에서 한도에 막혀 답답했던 분이라면, Max를 한 달만 써보세요. 한도 걱정 없이 Claude Code를 제대로 돌려보면 <strong>"이게 이렇게까지 되는 거였어?"</strong>라는 순간이 옵니다. 그 경험을 해보고 나서 Pro로 돌아갈지 판단해도 늦지 않습니다.

### 플랜 선택 가이드

| 상황 | 추천 플랜 |
|------|----------|
| "Claude Code가 뭔지 한번 써보고 싶다" | Pro $20로 체험 |
| "본격적으로 업무/프로젝트에 쓰고 싶다" | **Max $100 (강력 추천)** |
| "매일 하루 종일 사용한다" | Max $200 |

> **Pro $20은 "체험판"으로 생각하세요.** Claude Code의 잠재력을 확인하는 용도로는 충분합니다. 하지만 실제 생산성을 내려면 Max 요금제가 필요합니다.
